# Cloudflared Tunnel

这是一个面向 Home Assistant OS 的 `cloudflared` 加载项文档。  
它的目标是让你用最少的配置，把 Home Assistant 或宿主机上的其他本地服务通过 Cloudflare Tunnel 暴露出去。

This document describes the `cloudflared` add-on for Home Assistant OS.  
Its goal is to help you expose Home Assistant or other host-local services through Cloudflare Tunnel with minimal setup.

## 快速开始

1. 把本仓库添加到 Home Assistant 的自定义加载项仓库。
2. 安装 `Cloudflared Tunnel`。
3. 把 Cloudflare Dashboard 生成的 `Tunnel Token` 填进配置。
4. 如果你只想先暴露 Home Assistant，Cloudflare Dashboard 里的 `Service` 先填：

```text
http://127.0.0.1:8123
```

5. 启动 add-on。
6. 观察日志，确认出现 `Registered tunnel connection`。
7. 如果外网访问返回 `400 Bad Request`，再考虑开启 `auto_trust_local_proxy`。

## Quick Start

1. Add this repository to the Home Assistant add-on store.
2. Install `Cloudflared Tunnel`.
3. Paste the `Tunnel Token` generated by Cloudflare Dashboard.
4. If you only want to expose Home Assistant first, set the Cloudflare `Service` to:

```text
http://127.0.0.1:8123
```

5. Start the add-on.
6. Check the logs for `Registered tunnel connection`.
7. If public access returns `400 Bad Request`, then consider enabling `auto_trust_local_proxy`.

## 配置项

```yaml
tunnel_token: ""
log_level: info
metrics_enabled: false
metrics_port: 49312
additional_args: ""
auto_trust_local_proxy: false
auto_restart_core_after_proxy_patch: false
additional_trusted_proxies: ""
```

- `tunnel_token`：Cloudflare 远程托管 Tunnel 的 token，必填。
- `log_level`：`cloudflared` 日志级别。
- `metrics_enabled`：是否启用 `cloudflared` metrics 接口。
- `metrics_port`：启用 metrics 时使用的端口。
- `additional_args`：追加到底层 `cloudflared` 命令的参数。
- `auto_trust_local_proxy`：是否自动修复 Home Assistant 中的 `trusted_proxies`。
- `auto_restart_core_after_proxy_patch`：自动修复后是否自动重启 Home Assistant Core。
- `additional_trusted_proxies`：额外追加的信任代理地址，多个值用逗号分隔。
- 自动修复会尝试兼容不同 Supervisor / HAOS 版本下的配置挂载路径，例如 `/config`、`/homeassistant`、`/homeassistant_config`。

## Options

- `tunnel_token`: Token for a remotely managed Cloudflare Tunnel.
- `log_level`: Logging level used by `cloudflared`.
- `metrics_enabled`: Enable the `cloudflared` metrics endpoint.
- `metrics_port`: Port used for metrics when enabled.
- `additional_args`: Extra arguments appended to the `cloudflared` command.
- `auto_trust_local_proxy`: Automatically patch `trusted_proxies` in Home Assistant.
- `auto_restart_core_after_proxy_patch`: Restart Home Assistant Core after a proxy config change.
- `additional_trusted_proxies`: Extra trusted proxy addresses, separated by commas.
- Auto patching attempts to detect multiple Home Assistant config mount paths such as `/config`, `/homeassistant`, and `/homeassistant_config`.

## 重点提醒：`400 Bad Request`

如果你通过 Cloudflare 域名访问 Home Assistant 时看到：

```text
400: Bad Request
```

大多数情况下，并不是 Tunnel 没连上，而是 Home Assistant 不信任当前反向代理。

在某些 HAOS 发行版或宿主机网络布局下，特别是冬瓜 HAOS 或类似系统里，Home Assistant 看到的代理来源很可能不是 Docker 内网，而是：

- `127.0.0.1`
- `::1`

这也是本 add-on 提供 `auto_trust_local_proxy` 开关的核心原因。

## Important: `400 Bad Request`

If you get:

```text
400: Bad Request
```

when accessing Home Assistant through Cloudflare, the tunnel is often working correctly. The usual problem is that Home Assistant does not trust the reverse proxy.

In some HAOS variants and host networking layouts, especially on builds similar to Waxgourd HAOS, Home Assistant may see the proxy source as:

- `127.0.0.1`
- `::1`

This is the main reason why this add-on provides `auto_trust_local_proxy`.

## 推荐做法

- 默认保持 `auto_trust_local_proxy: false`
- 先确认 Tunnel 已连接，再排查 `400`
- 如果你已经确认外网返回 `400 Bad Request`，再开启 `auto_trust_local_proxy`
- 如果开启自动修复但不自动重启 Core，记得手动重启 Home Assistant

开启后，本 add-on 会确保 `configuration.yaml` 中至少有：

```yaml
http:
  use_x_forwarded_for: true
  trusted_proxies:
    - 127.0.0.1
    - ::1
```

如果你还需要额外信任其他代理地址，可以通过 `additional_trusted_proxies` 追加。

## Recommended Usage

- Keep `auto_trust_local_proxy: false` by default
- Confirm that the tunnel is connected before debugging a `400`
- Enable `auto_trust_local_proxy` only after you confirm public access fails with `400 Bad Request`
- If automatic patching is enabled without automatic restart, restart Home Assistant Core manually

When enabled, the add-on ensures that `configuration.yaml` contains at least:

```yaml
http:
  use_x_forwarded_for: true
  trusted_proxies:
    - 127.0.0.1
    - ::1
```

Use `additional_trusted_proxies` if you need more proxy addresses.

## Cloudflare Dashboard 怎么填

这个 add-on 不会替你创建 Public Hostname，也不会替你改 Cloudflare 路由。  
公网域名、路径规则、源站地址仍然需要你在 Cloudflare Dashboard 中维护。

常见 `Service` 填法：

- `http://127.0.0.1:8123`：Home Assistant
- `http://127.0.0.1:5000`：宿主机上的其他本地 Web 服务
- `http://192.168.1.10:8080`：监听在局域网地址上的宿主机服务

如果目标服务不监听在 `127.0.0.1`，就填它实际监听的地址，不要死写回环地址。

## Cloudflare Dashboard Configuration

This add-on does not create Public Hostnames or ingress rules for you.  
You still manage public hostnames and origin services in Cloudflare Dashboard.

Typical `Service` examples:

- `http://127.0.0.1:8123` for Home Assistant
- `http://127.0.0.1:5000` for another local host service
- `http://192.168.1.10:8080` for a service listening on a LAN address

If the target service does not listen on `127.0.0.1`, use the actual host address it binds to.

## 国内用户建议

- 如果你所在网络环境里 `QUIC/UDP` 不稳定，优先尝试：

```yaml
additional_args: "--protocol http2"
```

- 如果外网能连上 Cloudflare，但访问 HA 返回 `400`，优先排查 `trusted_proxies`
- 如果你使用的是国产盒子、冬瓜 HAOS、或者有宿主机网络包装，尤其要关注 `127.0.0.1` / `::1`
- 如果你后续还想暴露 `8124`、`7681` 这类管理端口，强烈建议加上 Cloudflare Access

## Recommendation for Mainland China Users

- If `QUIC/UDP` is unstable in your network, try:

```yaml
additional_args: "--protocol http2"
```

- If Cloudflare connects but Home Assistant returns `400`, inspect `trusted_proxies` first
- If you are using TV boxes, Waxgourd HAOS, or host-wrapped networking, pay extra attention to `127.0.0.1` / `::1`
- If you later expose management ports such as `8124` or `7681`, use Cloudflare Access

## 已知行为

- 本 add-on 使用 `host_network: true`，因此可以直接访问宿主机本地服务
- 自动修复 `trusted_proxies` 属于高影响行为，建议按需开启
- 自动修复后若不重启 Core，新配置不会立刻生效

## Known Behavior

- This add-on uses `host_network: true`, so it can directly reach host-local services
- Automatic `trusted_proxies` patching is a high-impact behavior and should be enabled deliberately
- If Core is not restarted after patching, the new configuration will not apply immediately

## 故障排查

- 启动时报缺少 token：确认 `tunnel_token` 已填写并保存
- 外网访问返回 `400 Bad Request`：优先检查 `trusted_proxies`，尤其是 `127.0.0.1` 和 `::1`
- Tunnel 已连接但服务打不开：检查 Cloudflare Dashboard 中的 `Service` 地址是否正确
- `QUIC` 不稳定或频繁超时：尝试 `additional_args: "--protocol http2"`
- metrics 端口冲突：调整 `metrics_port`

## Troubleshooting

- Missing token at startup: verify `tunnel_token` is filled and saved
- `400 Bad Request` from the public URL: inspect `trusted_proxies`, especially `127.0.0.1` and `::1`
- Tunnel connected but service unavailable: verify the `Service` address in Cloudflare Dashboard
- QUIC instability or timeouts: try `additional_args: "--protocol http2"`
- Metrics port conflict: change `metrics_port`

## 安全提示

`host_network: true` 会让这个 add-on 直接访问宿主机网络栈。  
请只暴露你明确需要公开的服务，并优先结合 Cloudflare Access、WAF 或其他身份验证机制使用。

## Security Notice

`host_network: true` gives this add-on direct access to the host network stack.  
Only expose services you intentionally want to publish, and prefer Cloudflare Access, WAF, or other authentication controls.
